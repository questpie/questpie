# Functions

Functions are used for custom operations that do not fit plain CRUD.

## Pattern

1. Create a file in your `functions/` directory
2. Export a default `fn({ schema, handler })` 
3. Functions are automatically composed via codegen and available on `app.functions`

Nested folders create nested routes:
- `functions/search.ts` → `app.functions.search`
- `functions/admin/stats.ts` → `app.functions.admin.stats`

## Example

```ts
// functions/get-revenue-stats.ts
import { fn } from "questpie";
import { z } from "zod";

export default fn({
  schema: z.object({
    period: z.enum(["day", "week", "month"]),
  }),
  handler: async ({ input, app, session }) => {
    const appointments = await app.api.collections.appointments.find({
      where: {
        scheduledAt: {
          gte: new Date(input.period),
        },
      },
    });
    return { appointmentCount: appointments.docs.length };
  },
  access: ({ session }) => !!session,
});
```

## Route Handler

The route handler no longer needs a separate `rpc` option — functions are included automatically:

```ts
const handler = createFetchHandler(app, { basePath: "/api" });
```

## Client Usage

```ts
const stats = await client.functions.getRevenueStats({
  period: "month",
});
```

## Access Control

Functions support an `access` callback to restrict who can call them:

```ts
export default fn({
  schema: z.object({ id: z.string() }),
  handler: async ({ input, app }) => {
    return app.api.collections.posts.findOne({ where: { id: input.id } });
  },
  access: ({ session }) => session?.user.role === "admin",
});
```

## When to Use Functions

- Booking orchestration
- Analytics aggregation
- External API workflows
- Multi-step domain logic
