# Type-Safe Contexts

Every server callback in QUESTPIE — hooks, access rules, functions, jobs, email handlers — receives a flat, fully typed context object. This is powered by `AppContext` module augmentation, generated automatically by `questpie generate`.

## How It Works

When you run `questpie generate`, the codegen produces a `.generated/index.ts` file that augments the `AppContext` interface:

```ts
// .generated/index.ts (auto-generated — do not edit)
declare module "questpie" {
  interface AppContext {
    db: /* typed Drizzle client */;
    collections: /* typed collection CRUD APIs */;
    globals: /* typed global CRUD APIs */;
    queue: /* typed job queue */;
    email: /* typed email service */;
    session: /* typed auth session */;
    storage: /* storage service */;
    kv: /* key-value store */;
    logger: /* logger */;
    // ...user services
  }
}
```

This means every handler automatically has typed access to the full app context — no manual type assertions needed.

## Basic Usage

Destructure what you need from any handler context:

```ts
// collections/appointments.ts
import { collection } from "#questpie";

export const appointments = collection("appointments")
  .fields(({ f }) => ({
    customer: f.relation({ to: "user", required: true }),
    scheduledAt: f.datetime({ required: true }),
    status: f.select({
      options: [
        { value: "pending", label: { en: "Pending" } },
        { value: "confirmed", label: { en: "Confirmed" } },
      ],
      default: "pending",
      required: true,
    }),
  }))
  .hooks({
    afterChange: async ({ data, operation, queue }) => {
      // queue is typed — sendAppointmentConfirmation is autocompleted
      if (operation === "create") {
        await queue.sendAppointmentConfirmation.publish({
          appointmentId: data.id,
        });
      }
    },
  })
  .access({
    update: ({ session }) => {
      // session is typed — user fields are autocompleted
      return !!session?.user;
    },
  });
```

## Context Properties

Every handler receives these properties from `AppContext`:

| Property | Type | Description |
|----------|------|-------------|
| `db` | `DrizzleClient` | Typed database client (may be a transaction inside hooks) |
| `collections` | `AppCollections` | Typed collection CRUD APIs (`collections.posts.find(...)`) |
| `globals` | `AppGlobals` | Typed global CRUD APIs (`globals.siteSettings.get()`) |
| `queue` | `QueueClient<AppJobs>` | Typed job queue (`queue.sendEmail.publish(...)`) |
| `email` | `MailerService<AppEmailTemplates>` | Typed email service |
| `session` | `Session \| null` | Authenticated user session |
| `storage` | `StorageService` | File storage service |
| `kv` | `KVStore` | Key-value store |
| `logger` | `Logger` | Logger instance |
| `search` | `SearchService` | Full-text search service |
| `realtime` | `RealtimeService` | Realtime event service |
| `t` | `(key, params?, locale?) => string` | Translation function |
| *custom services* | varies | Any services defined via `defineService()` |

## Context Across Surfaces

The same `AppContext` is available in all server callbacks:

| Surface | Context shape |
|---------|---------------|
| Collection hooks | `{ data, operation, original, session, db, collections, queue, ... }` |
| Collection access | `{ session, data, input, db, collections, ... }` |
| Global hooks/access | Same as collection variants |
| Functions (`fn()`) | `{ input, session, db, collections, queue, email, ... }` |
| Jobs (`job()`) | `{ payload, db, collections, queue, email, ... }` |
| Email handlers | `{ input, db, collections, ... }` |
| Dashboard loaders | `{ db, collections, session, locale, ... }` |
| Admin action handlers | `{ data, itemId, itemIds, session, db, collections, ... }` |

## Functions

Functions receive the full `AppContext` alongside their validated `input`:

```ts
// functions/create-booking.ts
import { fn } from "questpie";
import { z } from "zod";

export default fn({
  schema: z.object({ barberId: z.string(), serviceId: z.string() }),
  handler: async ({ input, collections }) => {
    // collections is typed — services, appointments are autocompleted
    const service = await collections.services.findOne({
      where: { id: input.serviceId },
    });
    if (!service) throw new Error("Service not found");

    return collections.appointments.create({
      barber: input.barberId,
      service: input.serviceId,
      scheduledAt: new Date(),
      status: "pending",
    });
  },
});
```

## Jobs

Jobs receive `AppContext` with their validated `payload`:

```ts
// jobs/send-appointment-confirmation.ts
import { job } from "questpie";
import { z } from "zod";

export default job({
  name: "send-appointment-confirmation",
  schema: z.object({ appointmentId: z.string() }),
  handler: async ({ payload, collections, email }) => {
    const appointment = await collections.appointments.findOne({
      where: { id: payload.appointmentId },
      with: { barber: true, service: true },
    });
    if (!appointment) return;

    await email.sendTemplate({
      template: "appointmentConfirmation",
      to: "customer@example.com",
      input: { appointment },
    });
  },
});
```

## Custom Services

Define custom services with `defineService()` — they become part of `AppContext`:

```ts
// services/blog.ts
import { defineService } from "questpie";

export default defineService(({ collections, db }) => ({
  async getPublishedPosts(limit = 10) {
    return collections.blogPosts.find({
      where: { status: "published" },
      limit,
      orderBy: { createdAt: "desc" },
    });
  },

  async getFeaturedPosts() {
    return collections.blogPosts.find({
      where: { featured: true },
    });
  },
}));
```

After codegen, `blog` is available on the context:

```ts
// In any handler
afterChange: async ({ blog }) => {
  const featured = await blog.getFeaturedPosts();
}
```

## AppConfig for Client

For client-side type safety, import `AppConfig` from the generated code:

```ts
// src/lib/client.ts
import { createClient } from "questpie/client";
import type { AppConfig } from "#questpie";

export const client = createClient<AppConfig>({
  baseUrl: "/api",
});
```

`AppConfig` is a flat type containing `collections`, `globals`, and `auth` — just what the client needs.

## Implicit Context with getContext()

All CRUD operations run within a context scope. Use `getContext()` to access context from reusable functions:

```ts
import { getContext } from "questpie";

// Reusable function — no need to pass context explicitly
async function logActivity(action: string) {
  const { db, session, locale } = getContext();

  await db.insert(activityLog).values({
    userId: session?.user.id,
    action,
    locale,
  });
}

// Usage in hook — works because CRUD runs within context scope
export const posts = collection("posts").hooks({
  afterChange: async () => {
    await logActivity("post_updated");
  },
});
```

### Safe Context Access

Use `tryGetContext()` when context might not be available:

```ts
import { tryGetContext } from "questpie";

function maybeLogUser() {
  const ctx = tryGetContext();
  if (ctx?.session) {
    console.log("User:", ctx.session.user.id);
  }
}
```

## Common Mistakes

- **Manually casting types** — With `AppContext` augmentation, you don't need `typedApp()`, `getApp()`, or type assertions. Everything is typed automatically.
- **Importing the runtime `app` object** — In handlers, destructure `collections`, `queue`, etc. from the context instead of importing the app instance.
- **Forgetting that `session` can be `null`** — Always check for `session` before accessing user properties.
- **Using generic queue access** — Use typed job-specific calls: `queue.jobName.publish(payload)`.
