# Project Structure

QUESTPIE doesn't enforce a specific file structure, but there's a recommended layout that makes the server-first ownership model clear. This structure scales from a single collection to hundreds.

## Recommended Layout

```text
src/
  questpie/
    server/
      questpie.config.ts        ← runtime config (db, adapters, secrets)
      modules.ts                ← module dependencies
      branding.ts               ← admin panel branding
      sidebar.ts                ← admin sidebar configuration
      dashboard.ts              ← admin dashboard widgets
      locale.ts                 ← content locale configuration
      admin-locale.ts           ← admin UI locale
      auth.ts                   ← authentication configuration
      collections/
        barbers.ts              ← collection definitions
        services.ts
        appointments.ts
        pages.ts
      globals/
        site-settings.ts        ← global definitions
      functions/
        create-booking.ts       ← function implementations
        get-active-barbers.ts
      jobs/
        send-confirmation.ts    ← background job definitions
      emails/
        appointment-confirmation.ts ← email templates
      blocks/
        hero.ts                 ← block type definitions
        cta.ts
      services/
        blog.ts                 ← custom services
      .generated/
        index.ts                ← codegen output (app instance + types)
        factories.ts            ← typed factories

    admin/
      builder.ts              ← qa() client builder
      fields/                 ← custom field renderers (if any)
      views/                  ← custom view components (if any)
      components/             ← custom UI components (if any)

  routes/
    api/
      app.ts                  ← mount platform fetch handler
    admin.tsx                 ← admin panel route

  migrations/
    0001-initial.ts           ← Drizzle migration files
```

## Why This Works

The layout enforces the WHAT vs HOW split:

| Directory | Responsibility | Defines |
|---|---|---|
| `questpie/server/` | **WHAT** — contracts and behavior | Schema, access, hooks, functions, jobs |
| `questpie/admin/` | **HOW** — rendering concerns | Branding, locale, custom renderers |
| `routes/` | **Mounting** — HTTP wiring | Route handlers, no business logic |

## File Responsibilities

### `questpie.config.ts`

The runtime configuration. This file provides database, adapters, secrets, and other runtime-only settings:

```ts
import { runtimeConfig } from "questpie";

export default runtimeConfig({
  db: { url: process.env.DATABASE_URL! },
  app: { url: process.env.APP_URL || "http://localhost:3000" },
  secret: process.env.BETTER_AUTH_SECRET || "dev-secret",
});
```

### `.generated/index.ts`

Auto-generated by `questpie generate`. This file imports all entities, creates the app instance, augments `AppContext`, and exports types. **Do not edit this file manually** — it is regenerated on every codegen run.

### `server/functions/`

Each file exports a function defined with `fn()`. Functions are automatically discovered by convention:

```ts
// server/functions/get-active-barbers.ts
import { fn } from "questpie";

export default fn({
  handler: async ({ collections }) => {
    return collections.barbers.find({ where: { isActive: true } });
  },
});
```

### `server/collections/`

One file per collection. Each file exports a single collection using the standalone factory:

```ts
import { collection } from "#questpie";

export const barbers = collection("barbers")
  .fields(...)
  .admin(...)
  .list(...)
  .form(...);
```

### `admin/builder.ts`

The client-side builder. It only handles registries and rendering concerns:

```ts
import { adminModule, qa } from "@questpie/admin/client";
import type { AppConfig } from "#questpie";

export const admin = qa<AppConfig>()
  .use(adminModule);
```

### `routes/api.ts`

Thin route handler. No business logic here:

```ts
import { createFetchHandler } from "questpie";
import { app } from "#questpie";

export const handler = createFetchHandler(app, {
  basePath: "/api",
});
```

## Scaling Patterns

### Large Collection Files

If a collection definition gets long (100+ lines), extract helper functions into the same file rather than splitting the definition across files:

```ts
// collections/barbers.ts
import { collection } from "#questpie";

function dayScheduleFields(f: FieldBuilder) {
  return {
    isOpen: f.boolean({ ... }),
    start: f.time({ ... }),
    end: f.time({ ... }),
  };
}

export const barbers = collection("barbers")
  .fields(({ f }) => ({
    name: f.text({ ... }),
    workingHours: f.object({
      fields: () => ({
        monday: f.object({ fields: dayScheduleFields }),
        tuesday: f.object({ fields: dayScheduleFields }),
        // ...
      }),
    }),
  }));
```

### Shared Field Patterns

For field patterns used across multiple collections (e.g., SEO fields, slug fields), create utility functions:

```ts
// server/utils/fields.ts
export function seoFields(f: FieldBuilder) {
  return {
    metaTitle: f.text({ label: { en: "Meta Title" }, maxLength: 255, localized: true }),
    metaDescription: f.textarea({ label: { en: "Meta Description" }, localized: true }),
  };
}
```

### Module Composition

Register modules in `modules.ts`:

```ts
// server/modules.ts
import { adminModule, auditModule } from "@questpie/admin/server";

export default [adminModule, auditModule] as const;
```

Module entities (collections, functions, jobs) are merged with your app's entities during codegen.

## Anti-Patterns

- **Putting schema rules in client code** — Field validation, access control, and hooks belong on the server. The client should only define how things render.
- **Splitting a collection across files** — Keep the entire `.collection().fields().admin().list().form()` chain in one file. It's the single source of truth for that entity.
- **Business logic in route handlers** — Route files should only mount handlers. Business logic goes in functions, hooks, or jobs.
- **Hardcoding view components** — Use the registry pattern for custom views. Hardcoded component references break the introspection contract.
