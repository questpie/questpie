Live Preview opens a fullscreen overlay with your frontend in an iframe so editors can see content updates as they work.

Two views ship out of the box:

- **`collection-form` (default)** — split-screen form on the left, iframe on the right. The iframe refreshes after save / autosave events. This is what every collection gets without further config.
- **`visual-edit-form` (opt-in)** — the [Visual Edit Workspace](./visual-edit): canvas iframe on the left, contextual right inspector. Edits land in the iframe via field-level patches without a save round-trip. Enable per collection with `.form(({ v }) => v.visualEditForm({}))`.

This page covers the default split-screen flow and the configuration both views share. For the patch-based workspace flow, see [Visual Edit Workspace](./visual-edit) and [Protocol & Reliability](./protocol).

## How It Works

1. Click the **eye icon** in the form toolbar
2. The admin opens a split-screen view:
   - **Left panel**: The full collection form (same fields, validation, actions)
   - **Right panel**: An iframe loading your frontend preview URL
3. On save, the preview iframe refreshes automatically

On mobile, the split view becomes **tabs** — switch between "Form" and "Preview".

## Configuring Preview

Add `.preview()` to your collection:

```ts title="collections/pages.ts"
import { collection } from "#questpie/factories";

export const pages = collection("pages")
	.preview({
		enabled: true,
		position: "right",
		defaultWidth: 50,
		url: ({ record }) => {
			const slug = record.slug as string;
			return slug === "home" ? "/?preview=true" : `/${slug}?preview=true`;
		},
	})
	.fields(({ f }) => ({
		title: f.text().required().localized(),
		slug: f.text().required(),
		content: f.blocks().localized(),
	}));
```

### Preview Options

| Option         | Type                            | Default   | Description                                                       |
| -------------- | ------------------------------- | --------- | ----------------------------------------------------------------- |
| `enabled`      | `boolean`                       | `true`    | Enable/disable preview                                            |
| `url`          | `(ctx) => string`               | —         | URL builder function (receives `{ record, locale? }`)             |
| `position`     | `"left" \| "right" \| "bottom"` | `"right"` | Preview panel position                                            |
| `defaultSize`  | `number`                        | `50`      | Default preview pane size as a percentage (0–100)                 |
| `minSize`      | `number`                        | `30`      | Minimum preview pane size as a percentage (0–100)                 |
| `defaultWidth` | `number`                        | —         | Alias for `defaultSize`. When both are present `defaultSize` wins |
| `minWidth`     | `number`                        | —         | Alias for `minSize`. When both are present `minSize` wins         |

## Draft Mode & Token Security

Preview URLs go through a token-based security flow:

1. Admin calls the `mintPreviewToken` route with the target path and TTL
2. Server returns an HMAC-SHA256 signed token
3. Admin iframe loads `/api/preview?token={token}`
4. Route handler verifies the token, sets a `__draft_mode` cookie, and redirects to the path
5. Your frontend checks the cookie and loads draft data if present

## Prepare A Frontend Page For Preview

A preview-ready frontend page needs three pieces: a collection preview URL, a loader that can read draft data, and preview markers around editable regions.

### 1. Configure The Collection

The collection must expose a URL that the admin can load in the iframe:

```ts title="collections/pages.ts"
export const pages = collection("pages")
	.preview({
		url: ({ record }) =>
			record.slug === "home"
				? "/?preview=true"
				: `/${record.slug}?preview=true`,
	})
	.form(({ v }) => v.visualEditForm({}));
```

Use `v.collectionForm({})` when you want the standard split-screen form. Use `v.visualEditForm({})` when you want the canvas + inspector workspace. Both views use the same `.preview()` URL.

### 2. Load Draft Data When The Preview Cookie Is Present

The `/api/preview` route sets `__draft_mode=true` before redirecting into your page. In the page loader, check that cookie and relax your public filters only for preview requests:

```ts title="lib/get-page.ts"
import { isDraftMode } from "@questpie/admin/shared";

export async function getPage({ slug, cookie }: {
	slug: string;
	cookie?: string;
}) {
	const isDraft = isDraftMode(cookie);
	const ctx = await createRequestContext();

	const page = await app.collections.pages.findOne(
		{
			where: { slug },
			...(isDraft ? {} : { stage: "published" }),
		},
		ctx,
	);

	return page;
}
```

For workflow collections, public routes should keep `stage: "published"` outside preview. For simpler collections, keep whatever public visibility filter your site already uses and remove it only when `isDraft` is true.

### 3. Wrap The Page With Preview Helpers

Pass the loaded record into `useCollectionPreview`, wrap the page in `PreviewProvider`, and mark editable regions with `PreviewField`:

```tsx title="routes/pages/$slug.tsx"
import {
	PreviewField,
	PreviewProvider,
	useCollectionPreview,
} from "@questpie/admin/client";

function Page({ initialData }) {
	const preview = useCollectionPreview({
		initialData,
		onRefresh: () => router.invalidate(),
	});

	return (
		<PreviewProvider
			isPreviewMode={preview.isPreviewMode}
			focusedField={preview.focusedField}
			onFieldClick={preview.handleFieldClick}
		>
			<PreviewField field="title">
				<h1>{preview.data.title}</h1>
			</PreviewField>
		</PreviewProvider>
	);
}
```

For block content, render through `BlockRenderer` and pass `preview.handleBlockClick` when preview mode is active. Fields rendered inside block components can use `PreviewField`; `BlockScopeProvider` scopes their paths to the selected block automatically when they are rendered through the block renderer.

## Field Focus Sync

Clicking a field in the form highlights it in the preview, and vice versa. This uses a bidirectional `postMessage` protocol between the admin iframe and the frontend.

In day-to-day usage you don't write the wire payloads yourself — `useCollectionPreview` and `PreviewField` handle them — but the message shapes are stable and documented in [Protocol & Reliability](./protocol). The relevant types include `FOCUS_FIELD` (admin → preview), `FIELD_CLICKED` and `BLOCK_CLICKED` (preview → admin), and the patch / commit / resync messages used by the [Visual Edit Workspace](./visual-edit).

Both directions resolve a single canonical peer origin and validate `event.origin` on receipt — wildcards are never used. See the protocol page for the security model.

## Frontend Integration

Use the `useCollectionPreview` hook in your frontend pages to detect preview mode and react to admin interactions:

```tsx title="routes/pages/$slug.tsx"
import {
	PreviewField,
	PreviewProvider,
	useCollectionPreview,
} from "@questpie/admin/client";

function PageComponent({ initialData }) {
	const {
		data, // Current data (refreshes on admin save)
		isPreviewMode, // Whether running inside the preview iframe
		selectedBlockId, // Currently selected block
		focusedField, // Currently focused field path
		handleFieldClick, // Attach to clickable field elements
		handleBlockClick, // Attach to clickable block elements
	} = useCollectionPreview({
		initialData,
		onRefresh: () => router.invalidate(),
	});

	return (
		<PreviewProvider
			isPreviewMode={isPreviewMode}
			focusedField={focusedField}
			onFieldClick={handleFieldClick}
		>
			<PreviewField field="title" as="h1">
				{data.title}
			</PreviewField>
		</PreviewProvider>
	);
}
```

The hook:

- Detects preview mode via `window.self !== window.top`.
- Listens for `PREVIEW_REFRESH` messages and calls `onRefresh`.
- Applies `PATCH_BATCH` ops to a local draft store, seeds the draft from `INIT_SNAPSHOT`, swaps or drops the draft on `COMMIT`, resets state on `FULL_RESYNC`, updates focus on `SELECT_TARGET`, and accepts same-origin `NAVIGATE_PREVIEW` (via `location.replace` so the back-button stack stays clean). The merged data flows back through `data`; `isDraftActive` tells you whether the iframe is showing a local draft or the canonical record.
- Tracks `selectedBlockId` and `focusedField` from admin focus events (`FOCUS_FIELD`, `SELECT_BLOCK`, `SELECT_TARGET`).
- Provides click handlers that send `FIELD_CLICKED` / `BLOCK_CLICKED` messages back to the admin.

There is no protocol toggle on the iframe side — the same hook handles refresh and patch messages transparently. Opting a collection into the [Visual Edit Workspace](./visual-edit) does not require any changes to your frontend page.

## Exit Preview Mode

The "Exit Preview" button redirects to `/api/preview?disable=true`, which clears the draft mode cookie.
