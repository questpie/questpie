---

The **Visual Edit Workspace** is an alternative form view for collections that have live preview enabled. Instead of the legacy "form on the left, iframe on the right" layout that refreshes on save, the workspace lets editors click directly in the preview to focus a field, edit it in a contextual inspector, and see the change land in the iframe via field-level patches — no full reload.

It is purely opt-in. The default `collection-form` view is untouched; existing preview integrations keep working.

## When To Use It

Pick the workspace when:

- You want editors to click on a heading/image/block in the rendered page and edit it without scrolling around in a form.
- You want field-level updates (typing a title, toggling a boolean) to land in the preview without re-running the loader.
- You want save/revert/stage transitions to keep the preview perfectly in sync without ad-hoc refresh logic.

Stick with the default `collection-form` when:

- The collection has no live preview, or
- The form is small enough that the legacy split view feels right.

## Enabling It

Add `.preview()` to the collection (the workspace requires a preview URL builder), then opt in via the form view:

```ts title="collections/pages.ts"
import { collection } from "#questpie/factories";

export const pages = collection("pages")
	.fields(({ f }) => ({
		title: f.text().required().localized(),
		slug: f.text().required(),
		content: f.blocks().localized(),
	}))
	.admin(({ c }) => ({
		label: { en: "Pages" },
		icon: c.icon("ph:file-text"),
	}))
	.preview({
		enabled: true,
		url: ({ record }) => `/${record.slug}?preview=true`,
		defaultSize: 60,
		minSize: 30,
	})
	.form({ view: "visual-edit-form" });
```

The `view` key resolves through the standard view registry, so any project-level overrides work the same way they do for `collection-form`.

## Layout

| Region              | Default size | Notes                                                                                                                                                   |
| ------------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Canvas (left)**   | 68%          | Full preview iframe. Reuses `PreviewPane` and the same origin-strict postMessage transport as the legacy live preview.                                  |
| **Inspector (right)** | 32%        | Render-prop router. Selects the body based on the current `VisualEditSelection`. Mobile collapses both regions into tabs.                               |

The default sizes come from `PreviewConfig.defaultSize` / `minSize` (with `defaultWidth` / `minWidth` accepted as aliases). If both sets are present, the explicit `*Size` keys win.

## Selection Model

The inspector renders one of seven modes, picked from the active selection:

| Selection kind | Inspector body                                                                                                  |
| -------------- | --------------------------------------------------------------------------------------------------------------- |
| `idle`         | **Document** — non-visual fields, SEO/meta, slug, status. Defaults to `AutoFormFields` for the collection.      |
| `field`        | A single field rendered through the existing `FieldRenderer`.                                                   |
| `block`        | The selected block's fields plus duplicate / remove actions.                                                    |
| `block-field`  | A specific field inside a block, scoped to `${blocksPath}._values.${blockId}.${fieldName}`.                     |
| `relation`     | Single relation field — clicking the related record opens the standard `ResourceSheet` flow.                    |
| `array`        | Whole-array editor (reorder, add).                                                                              |
| `array-item`   | One item inside an array field, indexed.                                                                        |

Clicks in the canvas are routed through `mapPreviewClickToSelection` — the same helper used by the legacy `parsePreviewFieldPath` — so block paths, relation hints, and bare field paths all translate consistently.

## Patch Protocol (V2)

The workspace drives the preview iframe through the V2 postMessage protocol, not full reloads. See [Protocol & Reliability](./protocol) for the wire spec; the high-level flow is:

1. **Mount.** Iframe sends `PREVIEW_READY`. Admin replies with `INIT_SNAPSHOT` (the canonical record at the current locale/stage).
2. **Edit.** Each react-hook-form change becomes a `PATCH_BATCH` with a monotonic `seq` and a list of `set` / `remove` ops.
3. **Save.** Mutation success triggers `COMMIT` — the preview swaps its local draft for the saved snapshot.
4. **Revert / restore / transition.** Triggers `FULL_RESYNC` — the preview discards its draft and re-runs the loader.

Iframes built with `useCollectionPreview` get all of this for free: the hook keeps a local draft store, applies patches with `applyPatchBatch`, and exposes the merged data through `data` on the result.

### Minimal Patches (Recursive Diff)

`PATCH_BATCH` payloads are produced by `diffSnapshot` — a recursive diff that descends into plain objects but treats arrays as atomic. Editing `meta.seo.title` in the inspector emits a single `set meta.seo.title = "X"` op rather than replacing the whole `meta` subtree on the wire. Arrays send a full `set` (with the new array as the value) so block reorders, tag updates, and similar operations stay correct without per-index shuffling.

| Change                       | Wire op                                         |
| ---------------------------- | ----------------------------------------------- |
| Edit `title`                 | `{ op: "set", path: "title", value: "..." }`   |
| Edit `meta.seo.title`        | `{ op: "set", path: "meta.seo.title", value: "..." }` |
| Add `meta.kicker`            | `{ op: "set", path: "meta.kicker", value: "..." }` |
| Remove `meta.kicker`         | `{ op: "remove", path: "meta.kicker" }`         |
| Reorder `tags: ["a", "b"]` to `["b", "a"]` | `{ op: "set", path: "tags", value: ["b", "a"] }` |

## Per-Field Tuning (`visualEdit`)

Fields can opt into workspace-specific overrides without touching the legacy form view:

```ts
f.text({
	meta: {
		admin: {
			visualEdit: {
				patchStrategy: "patch", // "patch" | "refresh" | "deferred"
				hidden: ({ data }) => !data.published,
				group: "seo",
				order: 2,
			},
		},
	},
});
```

| Key             | Default                                  | Effect                                                                                                                                  |
| --------------- | ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| `inspector`     | `undefined`                              | Component reference rendered instead of the default field component.                                                                    |
| `patchStrategy` | `"patch"` for scalars, `"refresh"` for relations / uploads / blocks / slug / computed | How changes propagate to the preview. Override when a custom field has a known compute pattern. |
| `hidden`        | `false`                                  | Hide the field from the document inspector body. Canvas click targets still work.                                                       |
| `group`         | `BaseAdminMeta.group`                    | Pin the field to a specific group in the document inspector — workspace-only override.                                                  |
| `order`         | declaration order                        | Display order inside the resolved group.                                                                                                |

Fields without a `visualEdit` block fall back to the type-driven default; the resolver lives in `@questpie/admin/client` (`resolveVisualEditMeta`, `resolvePatchStrategy`).

### Grouped Document Inspector

The default Document inspector body uses `AutoFormFields`, so a collection's existing `.form({ fields: [...] })` layout (sections, tabs) carries straight into the workspace. When you want a flatter, group-driven view instead — useful when the workspace's inspector is doing most of the editing and the Document panel is just SEO/meta/status — pass `DocumentInspectorBody` to `renderDocument`:

```tsx title="custom-pages-form.tsx"
import {
	DocumentInspectorBody,
	VisualEditFormHost,
} from "@questpie/admin/client";

export default function CustomPagesForm(props) {
	return (
		<VisualEditFormHost
			{...props}
			renderDocument={() => <DocumentInspectorBody />}
		/>
	);
}
```

`DocumentInspectorBody` reads the fields off the active controller, resolves each one's `visualEdit.group` / `order` / `hidden` (with server metadata winning), and renders each group with the same `FieldRenderer` the rest of the inspector uses. Fields with `visualEdit.hidden === true` are dropped; reactive `hidden` handlers defer to the field component itself.

```ts title="collections/pages.ts"
export const pages = collection("pages")
	.fields(({ f }) => ({
		title: f.text().required().localized().meta({
			admin: { visualEdit: { group: "content", order: 1 } },
		}),
		slug: f.text().required().meta({
			admin: { visualEdit: { group: "url", order: 1 } },
		}),
		seoTitle: f.text().meta({
			admin: { visualEdit: { group: "seo", order: 1 } },
		}),
		seoDescription: f.textarea().meta({
			admin: { visualEdit: { group: "seo", order: 2 } },
		}),
		// No `visualEdit.group` — falls through to the default group
		status: f.select(["draft", "published"]).default("draft"),
	}))
	.preview({ url: ({ record }) => `/${record.slug}` })
	.form({ view: "visual-edit-form" });
```

When no field opts into a non-default group, group headers are skipped — the inspector renders a flat field list, matching the legacy "AutoFormFields without sections" appearance.

If you need the underlying grouping data without the rendered output (e.g. to drive a custom layout), reach for the pure helper `groupFieldsForDocument` from `@questpie/admin/client`.

## Anatomy

The workspace primitives are exposed from `@questpie/admin/client`:

```ts
import {
	VisualEditFormHost, // <- highest-level: controller + provider + bridge + workspace
	VisualEditWorkspace, // <- 2-pane shell with selection provider
	VisualEditWorkspaceContent, // <- shell without provider (host owns one)
	VisualInspectorPanel, // <- right-pane router
	BlockInspectorBody, // <- default renderBlock implementation
	DocumentInspectorBody, // <- opt-in grouped document body
	useVisualEdit, // <- selection state hook
	useVisualEditController, // <- ResourceFormController in workspace context
	useVisualEditPreviewBridge, // <- save -> COMMIT, revert -> FULL_RESYNC
	useFormToPreviewPatcher, // <- RHF -> PATCH_BATCH dispatcher
	mapPreviewClickToSelection,
	mapPreviewBlockClickToSelection,
	groupFieldsForDocument, // <- pure helper backing DocumentInspectorBody
	resolveVisualEditMeta,
	resolvePatchStrategy,
} from "@questpie/admin/client";
```

Most projects only need to register the view (`.form({ view: "visual-edit-form" })`); the host wires every primitive together with sensible defaults. Drop down to the lower-level pieces only when you need to host the workspace inside a custom layout.

## Migrating From Legacy Live Preview

The classic live preview keeps working. To migrate a collection:

1. Confirm `.preview({ url: ... })` is set.
2. Add `.form({ view: "visual-edit-form" })`.
3. (Optional) Add `visualEdit` blocks to per-field metadata for fields that need a custom patch strategy or grouping.

No frontend code changes required — `useCollectionPreview` already handles both V1 (`PREVIEW_REFRESH`) and V2 (`INIT_SNAPSHOT` + `PATCH_BATCH` + `COMMIT`) transparently.
