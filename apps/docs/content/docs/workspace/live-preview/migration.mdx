---
title: Migration Guide
description: Upgrade existing live-preview integrations to the patch-based Visual Edit Workspace.
---

V1 — the legacy `collection-form` view — refreshes the preview iframe after every save / autosave. V2 — the [Visual Edit Workspace](./visual-edit) — drives the iframe with field-level patches over `postMessage`, no save round-trip.

V1 keeps working. V2 is opt-in per collection. You can run both side by side during the migration.

## V1 vs V2 at a glance

| Aspect            | V1 (`collection-form`)                                     | V2 (`visual-edit-form`)                                                                    |
| ----------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Update flow       | edit → save/autosave → invalidate → reload iframe          | edit → `PATCH_BATCH` → apply in iframe                                                     |
| Latency           | ~500 ms – 2 s (network round-trip + render)                | < 50 ms (local patch apply)                                                                |
| Layout            | Form on the left, iframe on the right (same view)          | Canvas iframe on the left, contextual right inspector                                      |
| Selection         | Field-only via `<PreviewField>` clicks                     | Selection state machine (document, field, block, block-field, relation, array, array-item) |
| Per-field tuning  | Form layout config (`sections`, `tabs`, `sidebar`)         | Form layout config **plus** `visualEdit.group` / `order` / `hidden` / `inspector` / `patchStrategy` |
| Iframe code       | Reads `data` from `useCollectionPreview`, refresh on save  | **Same** — the hook handles V1 + V2 messages transparently                                 |
| Save / commit     | Save → server → invalidate query → iframe reloads          | Save → `COMMIT` message → iframe swaps draft for canonical, then `onRefresh`               |
| Iframe reload     | Standard navigate                                          | `PreviewPane` buffers + replays `INIT_SNAPSHOT`; bridge re-seeds with current form values  |

## Per-collection upgrade

For each collection that already has `.preview({ url: ... })`, switch the form view from `v.collectionForm` to `v.visualEditForm`:

```ts title="collections/pages.ts"
.preview({
  enabled: true,
  url: ({ record }) => `/${record.slug}?preview=true`,
})
.form(({ v, f }) =>
- v.collectionForm({
+ v.visualEditForm({
    sidebar: { /* ... */ },
    fields: [/* ... */],
  })
);
```

That's the entire opt-in. The workspace immediately replaces the legacy split-form view with the 2-pane canvas + inspector. Sidebar config (slug compute, status toggle) carries over unchanged.

## What changes for the iframe / frontend page

Nothing required. `useCollectionPreview` already handles every V2 message (`INIT_SNAPSHOT`, `PATCH_BATCH`, `COMMIT`, `FULL_RESYNC`, `SELECT_TARGET`, `NAVIGATE_PREVIEW`) alongside the legacy `PREVIEW_REFRESH`. Existing pages continue to work:

```tsx title="routes/pages/$slug.tsx"
import {
  PreviewField,
  PreviewProvider,
  useCollectionPreview,
} from "@questpie/admin/client";

function PageRoute({ initialData }) {
  const { data, isPreviewMode, focusedField, handleFieldClick, handleBlockClick } =
    useCollectionPreview({
      initialData,
      onRefresh: () => router.invalidate(),
    });

  // No additional setup for V2 — the hook applies patches to its
  // local draft store and exposes them via `data`.
  return (
    <PreviewProvider isPreviewMode={isPreviewMode} focusedField={focusedField} onFieldClick={handleFieldClick}>
      <PreviewField field="title" as="h1">{data.title}</PreviewField>
      {/* ... */}
    </PreviewProvider>
  );
}
```

A new field on the result, `isDraftActive`, lets V2-aware pages tell whether the iframe is showing a local draft (after `INIT_SNAPSHOT`) or the bare `initialData`.

## Field-level tuning (optional)

After opting a collection into the workspace, tune individual fields with the new `visualEdit` admin metadata. The defaults are usually correct; reach for these when you want to override:

```ts
title: f.text().required().localized()
  .set("admin", { visualEdit: { group: "content", order: 1 } }),

slug: f.text().required().inputOptional()
  .set("admin", { visualEdit: { group: "url", order: 1 } }),

content: f.blocks().localized()
  // Hide blocks from the document inspector — they're edited
  // in the canvas + block inspector.
  .set("admin", { visualEdit: { hidden: true } }),

metaTitle: f.text().localized()
  .set("admin", { visualEdit: { group: "seo", order: 1 } }),
```

| Key             | Effect                                                                                                                                                           |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `group`         | Pin the field to a group in the Document inspector. Any field setting `group` flips the document body to the grouped layout (`DocumentInspectorBody`).           |
| `order`         | Sort within the resolved group. Lower values come first; ties fall back to declaration order.                                                                    |
| `hidden`        | Drop the field from the Document inspector. Canvas click targets still work — useful when the field is exclusively edited via the preview.                      |
| `patchStrategy` | `"patch"` (default for scalars) sends `PATCH_BATCH`. `"refresh"` triggers `PREVIEW_REFRESH` instead — set this on slug-style fields whose value is server-derived. |
| `inspector`     | Component reference rendered inside the inspector instead of the default field component. Resolves at nested-path granularity for object fields.                 |

See [Visual Edit Workspace](./visual-edit) for the full reference.

## Save / autosave behaviour

Identical on the V2 side. Saves flow through the existing mutation API (`useResourceFormController.mutations.update`); the workspace's `useVisualEditPreviewBridge` listens for `mutation.isSuccess` and emits a `COMMIT` message so the iframe drops its draft for the canonical record.

`autoSave` config on the collection still works; the patcher just keeps the iframe in sync between autosave commits.

## Mid-edit iframe reloads

In V1, an iframe reload mid-edit lost any unsaved state — you'd see the canonical record again. The workspace fixes this: `PreviewPane` buffers the latest `INIT_SNAPSHOT`, replays it on every `PREVIEW_READY`, and the workspace bridge additionally re-seeds with **current form values** so unsaved edits survive a navigation, hard refresh, or `NAVIGATE_PREVIEW` round-trip.

## Running both side by side

Pick which collections to migrate per release. `view: "collection-form"` and `view: "visual-edit-form"` are independent registry entries — collections you haven't touched continue rendering the legacy form. Iframe code doesn't need a flag because `useCollectionPreview` accepts both protocols.

## Rolling back

Set `view` back to `"collection-form"` (or use `v.collectionForm`) on the collection. No iframe-side changes needed; the page continues to receive `PREVIEW_REFRESH` after save.
