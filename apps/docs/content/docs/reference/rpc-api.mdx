# Functions API

Functions provide type-safe server procedures callable from the client SDK. They are defined with `fn()` from `"questpie"`, validated with Zod schemas, and support both JSON and raw HTTP modes.

## Defining Functions

### `fn(definition)` — JSON Function

Validated input/output with automatic Zod schema enforcement.

```ts
import { fn } from "questpie";
import { z } from "zod";

export default fn({
  schema: z.object({
    barberId: z.string(),
    date: z.string(),
    serviceId: z.string().optional(),
  }),
  handler: async ({ input, collections }) => {
    const barber = await collections.barbers.findOne({
      where: { id: input.barberId },
    });
    // ... compute available slots
    return { slots: availableSlots };
  },
});
```

**Definition fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `schema` | `ZodType` | No | Zod schema for input validation |
| `handler` | `(ctx) => Promise<TOutput>` | Yes | Function implementation |
| `access` | `(ctx) => boolean \| Promise<boolean>` | No | Access control |
| `outputSchema` | `ZodType` | No | Zod schema for output validation |

### `fn(definition)` — Raw Function

Direct access to `Request` and `Response` for streaming, file downloads, or custom protocols.

```ts
import { fn } from "questpie";
import { z } from "zod";

export default fn({
  schema: z.object({ format: z.enum(["csv", "json"]) }),
  raw: true,
  handler: async ({ input, collections }) => {
    const csv = await generateCsv(collections);
    return new Response(csv, {
      headers: { "Content-Type": "text/csv" },
    });
  },
});
```

### Handler Context

Both function types receive the full `AppContext` plus validated input:

```ts
{
  input: TInput;           // Validated input (from schema)
  request: Request;        // Raw HTTP request
  session: Session | null; // Authenticated user
  locale: string;          // Current locale
  db: Database;            // Typed database client
  collections: AppCollections;  // Typed collection CRUD APIs
  globals: AppGlobals;     // Typed global CRUD APIs
  queue: QueueClient;      // Typed job queue
  email: MailerService;    // Typed email service
  kv: KVStore;             // Key-value store
  logger: Logger;          // Logger
  storage: StorageService; // File storage
  // ...any custom services
}
```

All properties are fully typed via `AppContext` augmentation — no manual type casting needed.

## File-Based Routing

Functions live in a `functions/` directory. Nested folders create nested routes automatically.

```
functions/
  create-booking.ts        → POST /api/functions/createBooking
  get-active-barbers.ts    → POST /api/functions/getActiveBarbers
  get-revenue-stats.ts     → POST /api/functions/getRevenueStats
```

Each file exports a default `fn()` definition. The filename (kebab-case) is converted to camelCase for the route name.

## Mounting Functions

Functions are discovered automatically by codegen. No separate `rpc` option needed:

```ts
import { createFetchHandler } from "questpie";
import { app } from "#questpie";

const handler = createFetchHandler(app, {
  basePath: "/api",
});
```

## Access Control

Functions support an `access` callback:

```ts
fn({
  schema: z.object({ id: z.string() }),
  access: ({ session }) => {
    if (!session) return false;
    return session.user.role === "admin";
  },
  handler: async ({ input, collections }) => {
    // Only admins reach here
  },
})
```

If `access` returns `false`, the function responds with `403 FORBIDDEN`.

## Client Invocation

### Via Client SDK

```ts
import { createClient } from "questpie/client";
import type { AppConfig } from "#questpie";

const client = createClient<AppConfig>({ baseUrl: "/api" });

// Type-safe call
const result = await client.functions.createBooking({
  barberId: "b1",
  serviceId: "s1",
  scheduledAt: new Date().toISOString(),
  customerName: "John",
  customerEmail: "john@example.com",
});
```

### Via HTTP

```
POST /api/functions/createBooking
Content-Type: application/json

{ "barberId": "b1", "serviceId": "s1", ... }
```

## Error Handling

- Invalid input → `400 BAD_REQUEST` with Zod field errors
- Access denied → `403 FORBIDDEN`
- Handler throws → `500 INTERNAL_SERVER_ERROR` (original error in dev mode)
- Function not found → `404 NOT_FOUND`
