# Admin Builder API

The `qa()` builder is the **client-side** entry point for configuring your admin interface. It registers rendering components (fields, views, widgets, blocks) and configures branding and locale.

<Callout type="info">
For server-side configuration (collections, dashboard, sidebar, actions), see the [Collections](/docs/admin/collections), [Dashboard](/docs/admin/dashboard), and [Sidebar](/docs/admin/sidebar) docs.
</Callout>

## Creating an Admin Builder

```typescript
import { qa, adminModule } from "@questpie/admin/client";
import type { AppCMS } from "./server/cms";

const admin = qa<AppCMS>()
  .use(adminModule)          // Built-in field/view/widget renderers
  .branding({ name: "My Admin" })
  .locale({ default: "en", supported: ["en", "sk"] })
  .messages(customMessages);
```

The generic `<AppCMS>` provides type inference for your backend collections, globals, and other definitions.

## Builder Methods

### `.use(module)`

Merge another admin builder or module into this one:

```typescript
import { adminModule } from "@questpie/admin/client";

const admin = qa()
  .use(adminModule)        // Built-in fields, views, widgets
  .use(myCustomModule);    // Your custom extensions
```

The merge strategy:
- **fields** — Spread (last wins on conflicts)
- **views** — Spread (last wins on conflicts)
- **widgets** — Spread (last wins on conflicts)
- **blocks** — Spread (last wins on conflicts)
- **translations** — Deep merged

### `.branding(config)`

Configure admin branding:

```typescript
.branding({
  name: "My CMS",              // Shown in sidebar
  logo: LogoComponent,         // React component
  favicon: "/favicon.ico",     // Optional
})
```

### `.locale(config)`

Configure admin UI localization:

```typescript
.locale({
  default: "en",
  supported: ["en", "sk", "de"],
})
```

### `.messages(translations)`

Add custom translation messages:

```typescript
.messages({
  en: {
    "custom.greeting": "Hello!",
    "collection.posts.title": "Blog Posts",
  },
  sk: {
    "custom.greeting": "Ahoj!",
    "collection.posts.title": "Blogové príspevky",
  },
})
```

### `.fields(registry)`

Register custom field type renderers:

```typescript
.fields({
  colorPicker: myColorPickerField,
  markdown: myMarkdownField,
})
```

See [Custom Fields](/docs/admin/extensibility/custom-fields) for details.

### `.views(registry)`

Register custom view type renderers:

```typescript
.views({
  kanban: myKanbanView,
  calendar: myCalendarView,
})
```

See [Custom Views](/docs/admin/extensibility/custom-views) for details.

### `.widgets(registry)`

Register custom dashboard widget renderers:

```typescript
.widgets({
  weather: myWeatherWidget,
  notifications: myNotificationsWidget,
})
```

See [Custom Widgets](/docs/admin/extensibility/custom-widgets) for details.

### `.blocks(registry)`

Register page builder blocks:

```typescript
import blocks from "./blocks";

.blocks(blocks)
```

See [Block System](/docs/admin/blocks) for details.

### `.collections(config)`

Register client-side collection UI overrides:

```typescript
.collections({
  posts: postsAdmin,
  users: usersAdmin,
})
```

<Callout type="info">
In most cases, collection configuration is done on the **server** via `.admin()`, `.list()`, `.form()`, and `.actions()`. Client-side collection configs are used for overrides or when extending the server config.
</Callout>

### `.globals(config)`

Register client-side global UI overrides:

```typescript
.globals({
  siteSettings: settingsAdmin,
})
```

### `.defaultViews(config)`

Configure default view types for collections:

```typescript
.defaultViews({
  list: "table",
  edit: "form",
})
```

## Why No `.build()` Method?

Unlike the backend `q()` builder, the admin `qa()` builder does NOT have a `.build()` method:

| Backend (`q`) | Admin (`qa`) |
|---------------|--------------|
| Creates runtime instances | Pure configuration |
| Generates Drizzle tables | Defines React components |
| Executes complex logic | State IS the config |
| Must call `.build()` | NO `.build()` needed |

```typescript
// Backend — .build() creates runtime
const cms = q().collections({...}).build({ db: {...} });

// Admin — state IS the config, pass directly
const admin = qa().use(adminModule).branding({...});
<AdminLayoutProvider admin={admin} ... />
```

## Mounting the Admin UI

Pass the builder directly to `AdminLayoutProvider`:

```typescript
import { AdminLayoutProvider } from "@questpie/admin/client";

<AdminLayoutProvider
  admin={admin}
  client={cmsClient}
  queryClient={queryClient}
  LinkComponent={Link}
  activeRoute={location.pathname}
  basePath="/admin"
>
  <Outlet />
</AdminLayoutProvider>
```

## Type Inference

For global type inference, declare the type registry:

```typescript
declare module "@questpie/admin/client" {
  interface AdminTypeRegistry {
    cms: AppCMS;
    admin: typeof admin;
  }
}
```

This enables:
- Autocomplete for collection names
- Type-safe navigation helpers
- Inferred field types in hooks
